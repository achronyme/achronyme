# Audit: Parser & Compiler Diagnostics

**Date**: 2026-03-06
**Scope**: Parser error reporting, compiler diagnostics, span tracking, warning system, structured output
**Baseline**: v0.1.0-beta.3 — 133 tests passing, 7 crates, ~39k LOC
**Reference**: Rust compiler (rustc) diagnostics as gold standard for comparison

---

## Summary

This audit evaluates the quality of the parser and compiler's diagnostic output — error messages, warnings, source context, and machine-readability — by comparing against rustc's established patterns. The goal is to identify concrete gaps and prioritize improvements that benefit both human developers and AI agents.

The current diagnostics are functional but minimal. Errors report line:col and a message, but lack source snippets, multi-error recovery, suggestions, and structured output. The compiler path loses span information entirely. There is no warning system for the VM path.

---

## Findings

### Section 1: Diagnostic Rendering

| ID | Severity | Component | Description |
|----|----------|-----------|-------------|
| DX-01 | **HIGH** | All phases | **No source snippet rendering.** Errors display `parse error at line 3, col 24: expected number` but never show the offending source line, underline (`^^^`), or surrounding context. Rustc renders multi-line snippets with colored spans, labels, and margin line numbers. Users must manually open the file and navigate to the line. |
| DX-02 | **HIGH** | All phases | **No unified diagnostic type.** Each phase has its own error enum (`ParseError`, `CompilerError`, `IrError`, `R1CSError`, `RuntimeError`) with different structures. No shared `Diagnostic` struct with severity, primary span, secondary labels, suggestions, and notes. This prevents a consistent rendering pipeline. |
| DX-03 | **HIGH** | `compiler/src/error.rs` | **CompilerError has no source spans.** The bytecode compiler's 12 error variants carry only string messages — no line, column, or byte offset. Source location is irrecoverably lost between parsing (which has spans) and error display. E.g., "undefined variable: x" with no location. |
| DX-04 | **MEDIUM** | `achronyme-parser/src/ast.rs:6-10` | **Spans are point-only, not ranges.** `Span { line, col }` marks where a token starts but not where it ends. Rendering underlines like `let x: u32 = "hello"` with `^^^^^^^ expected u32` requires knowing the end position. Tokens carry `byte_offset` but this is also start-only. |

### Section 2: Error Recovery

| ID | Severity | Component | Description |
|----|----------|-----------|-------------|
| DX-05 | **HIGH** | `achronyme-parser/src/parser/` | **Parser reports only the first error.** Uses early-return `Result<T, ParseError>` with no recovery. A file with 5 syntax errors requires 5 compile-fix cycles. Rustc recovers at synchronization points (`;`, `}`, `fn`, `let`) and can report 20+ errors per compilation. This is especially costly for AI agents that could batch-fix all errors in one pass. |
| DX-06 | **MEDIUM** | `compiler/src/codegen.rs` | **Compiler reports only the first error.** Same early-return pattern. Even if the parser collected multiple errors, the compiler would still bail on the first semantic error. |

### Section 3: Suggestions & Help

| ID | Severity | Component | Description |
|----|----------|-----------|-------------|
| DX-07 | **MEDIUM** | All phases | **No "did you mean?" suggestions.** When a variable or function name is misspelled, the error says only `undefined variable: foob`. Rustc computes Levenshtein distance against in-scope names and suggests `did you mean: foo?`. One exception exists: the chained comparison error in `exprs.rs:69` includes a usage hint. |
| DX-08 | **MEDIUM** | All phases | **No contextual help notes.** Errors don't include secondary information like "this function expects 2 arguments" or "defined here" pointing to the original declaration. Rustc attaches `note:` and `help:` labels with secondary spans. |

### Section 4: Warning System

| ID | Severity | Component | Description |
|----|----------|-----------|-------------|
| DX-09 | **MEDIUM** | VM path | **No warnings for VM compilation.** The circuit path has taint analysis warnings (unused inputs), but the VM bytecode compiler has zero warnings. Missing: unused variables, unused `mut` on immutable bindings, unreachable code after `return`/`break`, variable shadowing in same scope. Rustc emits these by default and they catch real bugs. |

### Section 5: Structured Output

| ID | Severity | Component | Description |
|----|----------|-----------|-------------|
| DX-10 | **MEDIUM** | CLI | **No machine-readable error output.** All diagnostics are plain text to stderr. No `--error-format=json` flag. IDEs, LSPs, and AI agents must parse human-readable text to extract error locations. Rustc's JSON output includes file, line, column, byte ranges, message, severity, suggestions, and error codes. This is a prerequisite for the v0.2.0 LSP roadmap item. |

### Section 6: Error Propagation

| ID | Severity | Component | Description |
|----|----------|-----------|-------------|
| DX-11 | **MEDIUM** | CLI (`cli/src/commands/`) | **anyhow erases structured errors.** All errors are wrapped in `anyhow::anyhow!("Compile error: {e}")` at the CLI boundary, collapsing structured error types into flat strings. A rendering pipeline needs access to the original `Diagnostic` struct with spans, severity, and labels. |
| DX-12 | **LOW** | `achronyme-parser/src/parser/mod.rs` | **Duplicate parse API.** `parse_program` returns `Result<_, String>` (loses line/col), `parse_program_with_errors` returns `Result<_, ParseError>`. The string-returning API shouldn't exist — it forces callers to discard location information. |

## Architecture Comparison: Achronyme vs Rustc

| Aspect | Achronyme (current) | Rustc |
|--------|---------------------|-------|
| Error format | `"parse error at line 3, col 24: msg"` | Multi-line snippet with colors, underlines, labels |
| Errors per compile | 1 (bail on first) | 20+ with recovery |
| Span type | `{ line, col }` (point) | `BytePos(start)..BytePos(end)` (range) |
| Suggestions | 1 (chained comparisons) | Levenshtein, contextual, machine-applicable |
| Warning system | Circuit taint only | 100+ lint categories |
| Structured output | None | `--error-format=json` |
| Error codes | None | `E0308`, `E0425`, etc. with `--explain` |
| Diagnostic type | Per-phase enums | Unified `Diagnostic` with `SubDiagnostic` |
| Color support | None | ANSI colors, `--color=auto/always/never` |

## Proposed Architecture: Unified Diagnostic System

```
Diagnostic {
    severity: Error | Warning | Note | Help,
    message: String,
    code: Option<String>,          // "E001", "W002"
    primary_span: SpanRange,       // byte_start..byte_end
    labels: Vec<Label>,            // secondary spans with messages
    suggestions: Vec<Suggestion>,  // machine-applicable fixes
    notes: Vec<String>,            // extra context
}

SpanRange {
    file: PathBuf,
    byte_start: usize,
    byte_end: usize,
    line_start: usize,
    col_start: usize,
    line_end: usize,
    col_end: usize,
}
```

All phases (parser, compiler, IR, runtime) produce `Vec<Diagnostic>`. A single renderer handles:
- Human-readable terminal output (with ANSI colors)
- JSON output (for LSP/agents)
- Plain text (no color, for CI logs)

## Prioritized Fix Roadmap

### P0 — Foundation (enables all other improvements)

1. **DX-04 + DX-02**: Implement `SpanRange` (byte ranges) and unified `Diagnostic` struct in a shared crate or module
2. **DX-01**: Build diagnostic renderer with source snippets, underlines, and optional ANSI color

### P1 — Error Recovery (biggest UX win for humans and agents)

3. **DX-05**: Parser error recovery with synchronization tokens
4. **DX-03**: Add span tracking to `CompilerError`

### P2 — Intelligence

5. **DX-09**: VM-path warnings (unused variables, unreachable code)
6. **DX-07**: Levenshtein-based "did you mean?" for identifiers

### P3 — Machine Interface (prerequisite for LSP v0.2.0)

7. **DX-10**: `--error-format=json` CLI flag
8. **DX-11**: Preserve structured diagnostics through CLI layer

### P4 — Polish

9. **DX-08**: Secondary labels and help notes on errors
10. **DX-06**: Compiler-level multi-error collection
11. **DX-12**: Remove `parse_program` string API

## Methodology

- **Parser analysis**: Read all parser source files (`achronyme-parser/src/`), catalogued every error path, analyzed span propagation from lexer through AST
- **Compiler analysis**: Read all compiler source files (`compiler/src/`), traced error creation and propagation, verified span availability at each stage
- **Error rendering**: Traced all `Display` implementations and CLI formatting to understand what users actually see
- **Rustc comparison**: Referenced rustc's diagnostic architecture (`rustc_errors::Diagnostic`, `rustc_span::Span`) as the target standard
- **Agent impact**: Evaluated how each deficiency affects AI coding agents that consume compiler output to auto-fix code
