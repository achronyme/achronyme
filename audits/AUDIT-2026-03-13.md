## BUGS (must fix before release)

### B1. `set_proven_boolean()` ausente en helpers de test Plonkish
- **File:** `compiler/tests/complex_circuits_test.rs:53-55` (`plonkish_verify`)
- **File:** `compiler/tests/plonkish_test.rs:7-19` (`compile_and_verify`)
- **Issue:** Ambos helpers crean `PlonkishCompiler::new()` sin llamar `set_proven_boolean(compute_proven_boolean(&program))`. El equivalente R1CS sí lo hace (líneas 34-38).
- **Impact:** 87+ tests Plonkish corren sin optimización de enforcement booleano. Los tests dan falsa confianza — pasarían incluso si el enforcement de booleanos estuviera roto.
- **Fix:** Agregar `compiler.set_proven_boolean(ir::passes::bool_prop::compute_proven_boolean(&program))` en ambos helpers.

### B2. `validate_plonkish_json()` no valida `lookup_tables`
- **File:** `constraints/src/plonkish_export.rs:127-175`
- **Issue:** El campo `lookup_tables` se exporta (línea 110) pero `validate_plonkish_json()` nunca lo verifica. JSON con `"lookup_tables": "BASURA"` pasa validación.
- **Impact:** Herramientas downstream asumen que JSON validado tiene `lookup_tables` como array. Crash o comportamiento indefinido al parsear.
- **Fix:** Agregar `root["lookup_tables"].as_array().ok_or("missing/invalid 'lookup_tables' array")?;`

### B3. Rotación fuera de límites retorna zero silenciosamente
- **File:** `constraints/src/plonkish.rs:84-90`
- **Issue:** `Expression::evaluate()` con `Cell(col, rotation)` donde `row + rotation` cae fuera de `[0, num_rows)` retorna `FieldElement::ZERO` sin error.
- **Impact:** Un prover malicioso podría construir un witness donde un gate referencia una celda fuera de límites; el verificador trata esa celda como zero, potencialmente satisfaciendo el gate cuando no debería. Bug de soundness.
- **Fix:** En `verify()`, rechazar cualquier gate cuyas rotaciones accedan filas fuera de `[0, num_rows)`. Opción: cambiar `evaluate()` para retornar `Result` en lugar de zero silencioso.

### B4. Fixed columns no inicializadas defaultean a zero
- **File:** `constraints/src/plonkish.rs:171-177`
- **Issue:** `AssignmentStore::get()` retorna `FieldElement::ZERO` para filas más allá de la longitud actual del vector de una columna. Esto aplica a columnas fixed (selectores).
- **Impact:** Si un selector fixed no fue explícitamente puesto a 0 en filas de padding, `get()` retorna 0 de todas formas — correcto por accidente. Pero si un selector debería ser 1 en una fila de padding y no se inicializó, el gate se desactiva silenciosamente. Puede enmascarar bugs de construcción de circuitos.
- **Fix:** En `verify()`, assert que todas las columnas fixed tengan exactamente `num_rows` elementos. Agregar padding explícito en la fase de finalización del compiler.

## WARNINGS (should fix)

### W1. Sin validación estructural de gates/copies/lookups en JSON
- **File:** `constraints/src/plonkish_export.rs:148-156`
- **Issue:** `validate_plonkish_json()` solo verifica que `gates`, `copies`, `lookups` son arrays. No valida estructura interna: un gate sin `name`/`poly`, un copy sin `left`/`right`, etc., pasan validación.
- **Fix:** Validar estructura de cada entry: gates tienen `name` (string) + `poly` (expression); copies tienen `left`/`right` con `column`+`row`; lookups tienen `name`+`input_exprs`+`table_exprs`.

### W2. Longitud de assignments no validada contra column counts
- **File:** `constraints/src/plonkish_export.rs:158-172`
- **Issue:** `num_advice=5` con `assignments.advice=[]` pasa validación. No se verifica que `assignments.advice.len() == num_advice`, ni que cada columna tenga `num_rows` valores.
- **Fix:** Agregar checks dimensionales en `validate_plonkish_json()`.

### W3. Field elements no validados en JSON export
- **File:** `constraints/src/plonkish_export.rs:16-18`
- **Issue:** Los field elements se serializan como strings decimales vía `to_decimal_string()`. El validador no verifica que sean decimales válidos ni que estén en rango `[0, p)` del campo BN254.
- **Fix:** Agregar `FieldElement::from_decimal_str(value)` check en validación.

### W4. Column bounds no validados en expresiones JSON
- **File:** `constraints/src/plonkish_export.rs:20-29`
- **Issue:** Expresiones con `column.index=999` o `row=2^63` pasan validación sin bounds check contra `num_advice`/`num_fixed`/`num_instance`/`num_rows`.
- **Fix:** Bounds-check column indices y row references durante validación.

### W5. Sin tests de integración CLI para `--plonkish-json`
- **File:** `cli/tests/circuit_test.rs`
- **Issue:** No hay test que ejecute `circuit_command` con `plonkish_json_path = Some(...)` y verifique que el archivo JSON se crea con contenido válido. Tampoco test con backend incorrecto.
- **Fix:** Agregar al menos 3 tests: plonkish+json con inputs, plonkish+json sin inputs, r1cs+json rechazado.

### W6. 6 tests de compilación sin verificación de constraints
- **File:** `compiler/tests/plonkish_test.rs:1525-1582`
- **Tests:** `test_plonkish_addition_deferred`, `test_plonkish_subtraction_deferred`, `test_plonkish_mul_by_constant_free`, `test_plonkish_div_constant_free`, `test_plonkish_let_binding`, `test_plonkish_negation_deferred`
- **Issue:** Solo verifican que `compile_ir()` no falla. No generan witness ni llaman `verify()`. Pasarían incluso si el compiler produce constraint systems inválidos.
- **Fix:** Agregar witness generation + `system.verify()` a cada test.

### W7. Tests de paridad no llaman `optimize()`
- **File:** `compiler/tests/plonkish_test.rs` (sección IR soundness, líneas 1779-1968)
- **Issue:** Los tests de paridad copiados de R1CS no ejecutan `ir::passes::optimize(&mut program)`. El pipeline real (CLI) siempre optimiza. Los tests podrían no detectar bugs que solo aparecen post-optimización.
- **Fix:** Agregar `optimize()` donde el equivalente R1CS lo tiene.

### W8. Docstring de `write_plonkish_json()` no advierte sobre witness
- **File:** `constraints/src/plonkish_export.rs:101-104`
- **Issue:** El export incluye advice columns (witness privado) en plaintext JSON. El docstring no advierte sobre esto. En ZK, esto rompe la propiedad de zero-knowledge si el JSON se comparte.
- **Fix:** Agregar `/// WARNING: includes private witness data (advice columns). Do not share in untrusted environments.`

## NITS (nice to have)

### N1. Validación de backend ocurre después de IR lowering
- **File:** `cli/src/commands/circuit.rs:97-107`
- **Issue:** `--plonkish-json` con backend incorrecto se rechaza después de compilar IR. Mover la validación antes de `IrLowering` evitaría trabajo innecesario.

### N2. Proof output paths hard-coded en CWD
- **File:** `cli/src/commands/circuit.rs:242-244`
- **Issue:** `proof.json`, `public.json`, `vkey.json` siempre se escriben en el directorio actual. Confuso si el usuario ejecuta desde distintos directorios.

### N3. `materialize()` usa panic en lugar de Result
- **File:** `compiler/src/plonkish_backend/primitives.rs:87-93`
- **Issue:** Recursión > 1000 causa `panic!` en lugar de retornar error. El usuario ve un backtrace.

### N4. Lookup activation tiene dos semánticas mezcladas
- **File:** `constraints/src/plonkish.rs:408-424`
- **Issue:** Si hay selector explícito, la fila es activa si `selector ≠ 0`. Si no hay selector, se usa heurística (skip filas donde todos los inputs son zero). Mezclar ambas semánticas es confuso.

### N5. Descripción de `--plonkish-json` en args es misleading
- **File:** `cli/src/args.rs:82`
- **Issue:** Dice "Export Plonkish circuit + witness to JSON" pero funciona sin `--inputs` (exporta sin witness). Debería decir "(+ witness if --inputs)".

### N6. `unwrap()` en serialización de proof JSON
- **File:** `cli/src/halo2_proof.rs:478,483,489,497`
- **Issue:** `serde_json::to_string_pretty().unwrap()` — panics si serialización falla (extremadamente improbable).

### N7. Test `test_plonkish_mux_*` sin negative test para flag no-booleano
- **File:** `compiler/tests/plonkish_test.rs:1683-1731`
- **Issue:** `mux_selects_first/second/complex_branches` solo prueban flags 0 y 1. No hay test que verifique que `flag=2` sea rechazado (if-else sí tiene este test: `test_plonkish_if_else_boolean_enforcement`).

### N8. Bit extraction asume field de 256 bits
- **File:** `compiler/src/plonkish_backend/witness.rs:83-98`
- **Issue:** `limb_idx >= 4` retorna 0 silenciosamente. Correcto para BN254 pero frágil si se agrega otro campo.
