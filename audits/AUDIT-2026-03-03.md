# Audit: Code Quality, Security & ZK Soundness

**Date**: 2026-03-03
**Scope**: Full codebase — ZK backend soundness, VM memory safety, field arithmetic, panic points, error handling
**Baseline**: 988+ workspace tests passing, 7 crates, ~39k LOC
**Verification pass**: All critical findings independently verified against source code. 1 false positive identified and removed (P-04).

---

## Summary

Five parallel audits were conducted covering: (1) ZK backend soundness, (2) VM & memory safety, (3) BN254 field arithmetic, (4) panic/unwrap points, and (5) error handling patterns. The most critical finding is a systemic soundness bug in the Plonkish backend where unconstrained advice cells allow a malicious prover to forge proofs. The R1CS backend is sound. The VM has resource exhaustion risks and GC issues. Field arithmetic is correct with one parsing bug.

A second verification pass confirmed all critical/high findings and identified 1 false positive (P-04: `pow_by_squaring` exp==0 — the function has an early return guard and the caller also guards against it).

## Findings

### Section 1: Plonkish Backend Soundness

| ID | Severity | File | Description |
|----|----------|------|-------------|
| ZK-01 | **CRITICAL** | `plonkish_backend/primitives.rs:195-231` | `emit_arith_row` does not constrain `col_c` when `c_cell = None`. The arithmetic gate is `s_arith * (a * b + c - d) = 0`. When `c_cell` is `None`, `col_c` is an advice column (prover-controlled) never constrained to zero. A malicious prover can set `c` to any value, adding an arbitrary bias to every multiplication. This breaks all Mul, And, Poseidon S-box, MDS multiplication, division, and negation constraints. **Proof of exploit**: for `assert_eq(a * b, c)` with a=2, b=3, the prover sets `col_c = 94`, giving `d = 2*3 + 94 = 100`. The verifier accepts `2 * 3 = 100`. |
| ZK-02 | **HIGH** | `plonkish_backend/gadgets.rs:120-145` | `emit_bool_check` does not constrain `col_c`. Uses `s*(a*b + c - d) = 0` with `a=b=d=cell`, giving `cell^2 + c - cell = 0`. Prover sets `c = cell - cell^2` for any value, defeating boolean enforcement. Affects: Not, And, Or, Assert, Mux condition. |
| ZK-03 | **MEDIUM** | `plonkish_backend/gadgets.rs:276-303` | `emit_is_zero` check_row does not constrain `col_c`. Constraint 2 should enforce `diff * eq = 0` but actually gives `diff * eq + c = 0` with free `c`. Combined with broken boolean enforcement (ZK-02), IsEq/IsNeq results can be non-boolean. |
| ZK-04 | **MEDIUM** | `plonkish_backend/primitives.rs:74-89` | `materialize_val(Constant)` leaves `col_a` and `col_b` unconstrained. Gate gives `d = a*b + fe` where prover controls `a` and `b`. Any materialized constant can be offset. |
| ZK-05 | **LOW** | `r1cs_backend.rs:324-373` | IsLt/IsLe without prior RangeCheck uses 252-bit range enforcement. Values in `[2^252, p)` fail the range check. Potential inconsistency between IR evaluator (full 254-bit) and circuit (252-bit). Design trade-off, not a bug. |
| ZK-06 | **LOW** | `witness_gen.rs:346-353` | Poseidon witness fill uses `debug_assert_eq` for wire count validation — compiled out in release. Mismatch would produce invalid witness silently in release (verifier still rejects). |

**Non-findings (verified correct):**
- R1CS IsEq/IsNeq gadget — standard IsZero construction, sound
- R1CS boolean enforcement `b * (1 - b) = 0` — correct
- R1CS range check via bit decomposition — correct
- R1CS division by zero — `x * x_inv = 1` is unsatisfiable when `x = 0`, correct
- Poseidon capacity variable constrained to zero — correct
- `bool_prop` analysis is conservative and correct
- R1CS `multiply_lcs` and `materialize_lc` — fully constraining

**Root cause**: All Plonkish findings share the same root cause — `emit_arith_row` and related functions don't zero-constrain unused advice columns. Fix: `constrain_constant(col_c, ZERO)` when `c_cell = None`, and similarly for `col_a`/`col_b` when unused.

### Section 2: VM & Memory Safety

| ID | Severity | File | Description |
|----|----------|------|-------------|
| VM-01 | **CRITICAL** | `machine/control.rs:72-92`, `machine/vm.rs:18` | No call frame depth limit. `frames: Vec<CallFrame>` grows without bound. Functions with `max_slots = 1` can recurse millions of times before `STACK_MAX` is hit. OOM on the host. Fix: `const MAX_FRAMES: usize = 4096;` check before push. |
| VM-02 | **CRITICAL** | `machine/gc.rs:43` | GC `mark_roots` copies all 65,536 stack slots as roots, including stale handles above the current frame. Stack slots are never cleared after function return. Silent memory leak proportional to call history. Fix: zero register window on Return, or root only `stack[0..high_water_mark]`. |
| VM-03 | **HIGH** | `machine/upvalue.rs:16,44,57,68` | Four `.unwrap()` calls on heap upvalue lookups. Panic (crash host) if GC freed a referenced upvalue or upvalue chain is corrupted. Fix: `.ok_or(RuntimeError::SystemError(...))`. |
| VM-04 | **HIGH** | `heap.rs:297-341` | GC trace accesses `arena.data` directly via `.data.get(handle)`, bypassing `Arena::get()` which checks `is_free()`. If a handle points to a freed+recycled slot, trace reads the new occupant's data as children of the old object. Fix: use `self.lists.get(handle)`. |
| VM-05 | **HIGH** | `machine/interpreter.rs:92-94` | Jump target not validated against chunk bounds. `dest = decode_bx(instruction)` used directly as `ip`. Malformed bytecode silently terminates function (falls through to `ip >= chunk.len()` pop) instead of erroring. Also affects JumpIfFalse and ForIter. Fix: bounds check. |
| VM-06 | **HIGH** | `value.rs:55-57` | `Value::int()` silently truncates values outside i60 range via `& PAYLOAD_MASK`. Public API — any caller passing out-of-range i64 gets silent data corruption. Fix: `debug_assert!` or return `Option<Value>`. |
| VM-07 | **MEDIUM** | `machine/interpreter.rs:304-308` | `LoadConst` uses `unwrap_or(Value::nil())` for invalid constant index. Masks bytecode corruption. Fix: `ok_or(RuntimeError::OutOfBounds(...))`. |
| VM-08 | **MEDIUM** | `heap.rs:584-590` | `import_strings` replaces entire string arena. Existing `Value::string(handle)` values now point to different strings or out-of-bounds. Must only be called before execution. Fix: document or validate. |
| VM-09 | **MEDIUM** | `value.rs:250-256` | `as_handle()` masks with `0xFFFFFFFF` but payload is 60 bits — bits 32-59 silently ignored for crafted values. Defense-in-depth issue. |
| VM-10 | **MEDIUM** | `heap.rs` | `bytes_allocated` doesn't track growth from `list.push()` / `map.insert()`. GC may fire late. Recount on sweep self-corrects but drift accumulates between cycles. |
| VM-11 | **MEDIUM** | `machine/arithmetic.rs:177` | Integer division truncates toward zero (Rust default), not Euclidean. `-7 / 2 = -3` instead of `-4`. Semantic correctness concern for ZK language. |
| VM-12 | **LOW** | `machine/interpreter.rs:37` | No instruction count limit / fuel mechanism. `while true {}` hangs host indefinitely. |
| VM-13 | **LOW** | `machine/interpreter.rs:441-442` | Odd-length `upvalue_info` causes panic on closure creation. `upvalue_info[i+1]` out of bounds. |
| VM-14 | **LOW** | `value.rs:141-143` | `is_obj()` includes `TAG_NATIVE` (10) but natives are not heap-allocated. Semantically misleading. |
| VM-15 | **LOW** | `machine/stack.rs:12-14` | `base + reg` addition not overflow-checked. Practically unreachable due to upstream bounds. |
| VM-16 | **LOW** | `heap.rs:547-553` | Map entry size inconsistency in `recount_live_bytes` (missing `u64` hash overhead). Minor accounting drift. |

**Positive observations:**
- Integer arithmetic uses `checked_add/sub/mul` with i60 range validation
- Stack register access is bounds-checked via `.get()`
- Invalid opcodes caught via `OpCode::from_u8() -> Option`
- Arena allocator has proper free-list semantics via public API
- GC uses iterative worklist (not recursive DFS) — no Rust stack overflow during tracing
- Arity checking present for both closures and native calls
- Stack is fixed-size `Box<[Value]>` — no reallocation

### Section 3: BN254 Field Arithmetic

| ID | Severity | File | Description |
|----|----------|------|-------------|
| FA-01 | **MEDIUM** | `field/parsing.rs:46-75` | `from_decimal_str` silently produces wrong results for inputs >= 2^256. Carry from multiply-by-10 and add-digit loops overflows past 4th limb and is discarded. E.g., parsing `2^256 + 1` produces `1` instead of the correct `(2^256 + 1) mod p`. The correct pattern (checking carry) already exists in `BigInt::from_decimal_str`. |
| FA-02 | **LOW** | `field/parsing.rs:78-138` | `from_hex_str` and `from_binary_str` silently reduce values >= p instead of rejecting. Inconsistent with `from_le_bytes` which rejects >= p by returning `None`. |
| FA-03 | **LOW** | `field/mod.rs:58-63` | `from_canonical` does not validate input range. Name implies canonical form [0, p) but accepts any `[u64; 4]`. Montgomery multiplication inherently reduces, so result is correct, but semantically misleading. |

**Verified correct:**
- MODULUS = BN254 Fr (scalar field, NOT base field Fq) — confirmed via Fermat primality test
- R = 2^256 mod p — correct
- R2 = R^2 mod p — correct
- INV: `MODULUS[0] * INV ≡ -1 (mod 2^64)` — correct
- P_MINUS_2 = p - 2 — correct for Fermat inversion
- Montgomery CIOS reduction — follows bellman/ff pattern, carry propagation correct
- `mac` worst case = u128::MAX (no overflow, zero headroom but correct)
- Constant-time discipline: `pow` always squares+multiplies with `ct_select`; `neg` uses `(x | -x) >> 63` idiom; `sub` uses branchless mask; `add` high limb sum fits in u64 (verified)
- BigInt arithmetic: overflow/underflow/div-by-zero all checked with `Result` types

### Section 5: Error Handling Patterns

| ID | Severity | Category | Description |
|----|----------|----------|-------------|
| EH-01 | **HIGH** | Missing trait impl | `CompilerError` and `RuntimeError` do not implement `Display` or `std::error::Error`. All user-facing errors formatted with `{:?}` (Debug), producing output like `ParseError("expected expression")`. |
| EH-02 | **HIGH** | Wrong classification | `From<String> for CompilerError` converts any string error to `CompilerError::ParseError`. Module loading errors, type errors, I/O errors all silently become "parse errors". |
| EH-03 | **MEDIUM** | Inconsistent formatting | CLI commands use `{:?}` (Debug) for errors (`run.rs`, `compile.rs`, `disassemble.rs`) while `circuit.rs` correctly uses `{e}` (Display). Users see Rust internals like `InvalidHeader([65, 67, 72, 0])`. |
| EH-04 | **MEDIUM** | Lost context | `ProveError` wraps only `String` in all variants, losing original error types (`IrError`, `R1CSError`, etc.). Same for `R1CSError::EvalError(String)`. |
| EH-05 | **MEDIUM** | Inconsistent handling | VM arithmetic uses 3 different error patterns: `.unwrap()` (int path), `.ok_or(RuntimeError::InvalidOperand)?` (bigint path), `.ok_or_else(|| RuntimeError::TypeMismatch(...))?` (field path). Same conceptual operation, different error styles. |
| EH-06 | **MEDIUM** | Duplicate API | `parse_program` returns `Result<_, String>` (loses line/column info), `parse_program_with_errors` returns `Result<_, ParseError>` (preserves it). First API shouldn't exist. |
| EH-07 | **LOW** | Swallowed error | VM debug symbol loading silently skips invalid UTF-8 names (`vm.rs:164`). |
| EH-08 | **LOW** | Swallowed error | `run.rs:59` — path canonicalize failure silently skipped, breaking circular-import detection for symlinked paths. |
| EH-09 | **LOW** | Internal leakage | Error messages expose internal concepts to users: "Dangling string handle", "bad field handle", Rust struct Debug formatting. |
| EH-10 | **LOW** | Missing Error impl | `WitnessError`, `ProveError`, `PlonkishError`, `BigIntError` have `Display` but no `std::error::Error` impl. |

## Prioritized Fix Roadmap

### P0 — Cryptographic Soundness (blocks any proof generation trust)
1. **ZK-01**: Constrain `col_c` to zero in `emit_arith_row` when `c_cell = None`
2. **ZK-02**: Constrain `col_c` to zero in `emit_bool_check`
3. **ZK-03**: Constrain `col_c` to zero in `emit_is_zero` check_row
4. **ZK-04**: Constrain `col_a`/`col_b` to zero in `materialize_val(Constant)`

### P1 — Host Safety (prevents crashes and resource exhaustion)
5. **VM-01**: Add `MAX_FRAMES` constant (4096) with check before push
6. **VM-03 / P-01**: Replace upvalue `.unwrap()` with `ok_or(RuntimeError::...)`
7. **VM-05**: Validate jump targets against chunk bounds
8. **P-05**: Change `LC::evaluate` from `panic!` to `Result`

### P2 — Correctness
9. **FA-01**: Fix `from_decimal_str` carry overflow (pattern exists in BigInt)
10. **VM-02**: Clear stack slots on function return (GC memory leak)
11. **VM-04**: Use `Arena::get()` in GC trace instead of direct `.data.get()`
12. **VM-06**: Add `debug_assert!` in `Value::int()` for i60 range

### P3 — Error Quality
13. **EH-01**: Implement `Display` + `std::error::Error` for `CompilerError` and `RuntimeError`
14. **EH-02**: Remove `From<String> for CompilerError`
15. **EH-03**: Standardize CLI error formatting to `{e}`

### P4 — Defense in Depth
16. **VM-07**: Replace `LoadConst` `unwrap_or(nil)` with error
17. **VM-12**: Add optional instruction fuel/budget
18. **VM-13**: Validate `upvalue_info.len() % 2 == 0`
19. **P-10/11/12**: Replace CLI `.expect()` with `anyhow` errors

## Methodology

- **ZK Soundness**: Manual review of constraint generation in both R1CS and Plonkish backends. Verified each gadget's constraint equations, checked for under-constrained variables, and analyzed exploit scenarios.
- **VM Safety**: Review of memory management (arena, heap, GC), value representation (NaN-boxing), interpreter loop, and all opcode handlers.
- **Field Arithmetic**: Verified all BN254 constants independently, audited Montgomery CIOS reduction, checked constant-time discipline, and tested edge cases in parsing.
- **Panic Points**: Grep for `.unwrap()`, `.expect()`, `panic!()`, `unreachable!()`, and unchecked indexing across all non-test production code (~80 sites analyzed).
- **Error Handling**: Review of all error types, their trait implementations, propagation patterns, and formatting at user-facing boundaries.
